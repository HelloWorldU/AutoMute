# DESIGN

> AutoMute 的顶层设计。由 `/create` 对齐会话收敛而来（2026-05-31）。

## 终点

看直播时（**看口型的内容**，画面里有人脸说话），从一条**已混好的音频**里
**实时、低延迟地掐掉某一个指定人的声音**，其余人和背景音原样播放。
平台 Windows。同时作为深入学 C++ 的载体。

## 核心信念

1. **终点即罗盘** — 从"最终要做出什么"反推，识别路径上的模糊点与技术风险。
2. **延迟是机制的物理代价，不是 bug** — 你没法掐掉一个还没听到的声音。
3. **C++ 自建有边界** — 把学习火力集中在 C++ 最擅长的实时音频管线，ML 借现成模型。

## 范围

| | |
|---|---|
| **In** | 单条混合音频流；轮流说话（不重叠）；事先录入目标声纹；实时识别并掐声 |
| **Out (v2+)** | 多人同时说话的源分离；在线自动聚类选人；视觉口型识别；跨平台 |

## 关键决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 音源 | 抓 Windows 系统/应用混合音频（WASAPI loopback） | 平台已混好，只有一条流 |
| 延迟架构 | **反应式**，缓冲≈0，不延迟 | 看口型场景下，音画不同步比"开头泄漏"更难受 |
| C++ 边界 | 实时音频引擎自写；声纹模型借 ECAPA/x-vector 轻量版，经 ONNX Runtime C++ 调用 | 造该造的轮子，借不该造的 |
| 认人方式 | **事先录一段目标声音 → 存声纹 → 实时比对** | 稳、快、路径清晰；在线聚类留给 v2 |

## 延迟模型

整条延迟 = 管线延迟 + 判断延迟：

| 组成 | 量级 | 可压缩性 |
|------|------|----------|
| 管线（抓取→处理→回放，WASAPI 低延迟） | ~10–30ms | 能榨干 |
| 判断（声纹认出"这是他"） | ~200–400ms | **物理墙**，低于 ~150ms 准确率断崖 |

**反应式架构的代价**：每次目标开口的头 200–400ms 会漏出来，之后被掐掉。
靠"压短判断窗口 + 快模型"让泄漏逐次收窄，而非追求零泄漏。

> 突破判断墙的杀手锏是**视觉口型检测**（看见他开口就知道），但那是 v2 杠杆。

## 技术风险

| 风险 | 缓解 |
|------|------|
| 开头泄漏（物理下限） | 压短判断窗口 + 快模型 |
| 背景音乐/噪声干扰识别 | 先过 VAD + 调相似度阈值 + 保证录入样本干净 |
| WASAPI 抓取 + 低延迟回灌 | milestone 1 先打通这条硬骨头 |

## C++ 自建边界

| 部分 | 自写 / 借用 | 学习价值 |
|------|------------|----------|
| 实时音频引擎（环形缓冲 / 无锁队列 / 低延迟线程 / DSP） | **自写** | C++ 主场，硬实时肌肉，拉满 |
| 声纹 embedding 模型 | 借 ONNX Runtime C++ API 调预训练模型 | 学第三方库集成 / FFI |
